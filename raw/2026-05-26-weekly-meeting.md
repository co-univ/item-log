## 1. 지난 기간 발생했던 이슈 공유

### 이슈

### 문의

#### [유저] 복습모드 풀이기록 조회 문의

- 실시간 풀이 진행 후 문제셋을 종료하고 복습모드로 전환하면, 각 문제마다 누가 풀었는지 목록을 볼 수 없다는 문의가 있었다.
- 이미 `2회차:WebSocket` 문제셋을 복습모드로 변경한 뒤라 실시간 종료 상태로 되돌릴 수 있는지 문의했다.
- DB에 문제를 푼 사람 목록이 남아 있는지, 또는 복습모드를 다시 실시간 종료 상태로 복원하는 방식이 더 쉬운지 확인이 필요하다고 문의했다.
- 목적은 어떤 문제가 잘 풀렸는지 분석하고, 누가 풀었는지 아카이브하며, 팀원 참여 여부를 확인하는 것이다.
- 후속 정리: 종료된 문제셋에서 maker, player별로 어떤 화면과 정보를 노출할지 동선을 정리한다.
- 현재 보이는 정보가 충분한지에 대해서는 충분하다고 판단했다.

#### 문제 셋 통계 관련

- 실시간 모드 문제셋을 풀고 로고 또는 탭 닫기로 나간 뒤 LNB를 통해 대시보드로 들어가는 흐름이 언급됐다.
- 해당 항목은 회의록에서 취소선 처리됐다.

#### [PLAYER] 완료된 문제 셋 -> 상세 대시보드 조회 조건

- player가 문제셋 풀이를 완료했지만 아직 모든 팀원이 풀지 않았거나, 관리자가 학습모드 문제셋을 종료하지 않은 상태를 논의했다.
- 이때도 player는 상세 대시보드를 볼 수 있다.
- 오답률만 실시간으로 업데이트된다.
- 후속으로 학습 모드 풀이 완료 후 동선만 정리한다.
- 피그마에는 통계확인 버튼 추가가 필요하다. 대상은 실시간 모드와 학습 모드의 player `풀이 완료` 상태다.

## 3. 기획 회의

### 등수정보: 퍼센트별 노출 기준/개수 정하기

- 피드백 기반으로 재논의하며, 백엔드 데이터 공유가 필요하다.
- 기본 룰은 정답률과 비례하고, 최대 영역의 40%를 사용하는 방향이다.
- 디자인 과정에서 기획/프론트 소통이 필요하다.
- 상위 N개 안에 정답이 없는 경우 정답 영역을 따로 두는 편이 나을지 논의가 필요하다.
- 메인 서버에 있는 문제 데이터를 기반으로 먼저 추출해야 한다.
  - 백엔드가 데이터를 전달한다.
  - 해당 데이터를 기반으로 와이어프레임을 재구성한 뒤 5월 31일 회의에서 추가 공유한다.
- 유형별 노출 기준도 정리한다.
  - 객관식 복수정답인 경우에는 그룹화해서 노출한다.

### 문제 카드 UI 개선점 논의

- 칸반 형식을 검토한다.
- 카드에는 문제셋 제목과 날짜 정보만 두고, 더보기 아이콘에 버튼 동작을 모두 넣는 안이 나왔다.
- 카테고리 값은 문제 카드에 노출하지 않는 방향으로 논의됐다.
- 상태에 따른 칸반 형태로 정리한다.
- 카테고리는 현재 미노출이나, UI 개선 시점에는 개선이 필요할 수 있다.
- `풀기`와 `관리`가 굳이 같은 디자인일 필요는 없다는 의견이 있었다.
  - 목적 자체가 다르다.
  - 풀기는 현재 디자인도 큰 문제가 없어 보인다.
  - 관리는 기능이 계속 추가되며 복잡해지고 있다.
  - 개발 관점에서는 동일한 코드에 다른 기능을 넣어야 해서 복잡하다.
- 복습에서도 풀이관리를 볼 수 있어도 되지 않을지 논의됐다.
- `문제관리`가 `팀관리` 하위로 들어가도 되지 않을지 논의됐다.
  - 찬성: 관리자 입장에서 봐야 하는 메뉴를 모아둘 수 있다.
  - 반대: 팀관리 하위 개념이 문제관리라고 보기는 어렵다.

## 6. AI의 요약

- 6월 2일 배포는 개인 워크스페이스와 풀이결과 대시보드 2차를 중심으로 알파 반영 일정을 맞추는 방향으로 정리했다. 일정상 일부 알림 항목은 6월 16일 추가 배포로 밀고, 5월 31일까지 알파 반영 후 6월 1일 전체 QA를 진행하기로 했다.
- 개인 워크스페이스는 회원 가입/기존 유저 자동 생성, 실시간 문제셋 생성 차단, 초대링크 차단 등 핵심 범위가 완료 체크됐다. 백엔드는 팀명 수정 API와 마이그레이션 잡, 기본 생성 워크스페이스 명칭을 닉네임 기준으로 바꾸는 작업을 확인했다.
- 복습모드 전환 후 문제별 제출자 목록을 볼 수 없다는 사용자 문의를 계기로, 종료된 문제셋에서 maker/player별로 어떤 정보와 동선을 제공할지 별도 정리가 필요하다고 판단했다. 단순 복구 요청이 아니라 참여율 모니터링과 사후 조회 요구까지 포함하는 문제로 보아 별도 후속 티켓으로 분리했다.
- 풀이결과 대시보드 2차에서는 오답률 산식을 `최초 제출 시 오답인 유저 / 해당 문제 제출 유저 수 * 100`으로 정하고 소수 둘째 자리에서 반올림하기로 했다. 본인 기준 카테고리별 정답률, 본인 문제셋별 풀이기록, 문제셋별 오답률, player의 풀이 완료 상태 통계확인 버튼이 함께 확인됐다.
- 등수정보 노출 기준은 실제 백엔드 데이터가 먼저 공유되어야 디자인을 다시 잡을 수 있다고 정리했다. 기본 방향은 정답률에 비례하고 최대 영역의 40%를 쓰는 것이지만, 상위 N개 안에 정답이 없는 경우와 객관식 복수정답 그룹화 같은 유형별 기준은 추가 명세가 필요하다.
- 문제 카드 UI는 상태 기반 칸반 형태, 제목/날짜 중심 카드, 더보기 메뉴 액션으로 단순화하는 안이 논의됐다. 카테고리는 현재 카드에 노출하지 않는 방향이지만 UI 개선 시 다시 검토할 수 있고, `문제관리`를 `팀관리` 하위로 넣는 안은 정보구조상 반대 의견이 있어 별도 정리가 필요하다.
