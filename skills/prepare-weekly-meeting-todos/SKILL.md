# Prepare Weekly Meeting Todos

## 목적

지난 회의의 약속과 Jira의 현재 상태를 함께 읽고, 다음 회의 전에 확인하거나 실행해야 할 일만 파트별로 상기시킨다. Jira 티켓을 나열하는 작업이 아니라 각 항목의 현재 맥락에서 필요한 **다음 행동**을 만드는 작업이다.

## 입력

다음 입력을 우선 사용한다.

- 지난주 회의록 Notion 페이지
- 할 일을 넣을 대상 회의록 Notion 페이지
- 회의록에서 언급된 Jira 키와 MAIT-PM 범위의 관련 티켓

링크가 제공되면 해당 페이지와 티켓을 직접 읽는다. 페이지 제목이나 일부 블록만 보고 판단하지 말고 토글의 하위 블록, Jira 설명, 댓글, 연결 티켓까지 가능한 범위에서 확인한다. 접근할 수 없는 입력이 있으면 추측해서 쓰지 말고 누락된 입력을 보고한다.

## 워크플로우

### 1. 대상 회의록 구조 확인

1. 대상 회의록에서 제목이 `2. 파트별`로 시작하는 섹션을 찾는다.
2. 그 하위의 파트별 토글 이름, 순서, 기존 체크박스를 기록한다.
3. 같은 Jira 키나 같은 행동을 가리키는 기존 항목을 미리 식별한다.

해당 섹션이나 토글이 없으면 임의의 구조를 만들지 않는다. 사용자에게 필요한 위치를 보고하고 쓰기를 중단한다.

### 2. 지난 회의에서 후속 행동 추출

지난 회의록을 끝까지 읽고 다음 항목을 원자 단위로 추출한다.

- 담당자가 하기로 한 일
- 다음 회의까지 확인하거나 결정하기로 한 일
- 기획, 디자인, 개발, QA 결과물이 필요한 미완료 논의
- Jira 티켓으로 이어진 결정이나 후속 작업
- 다른 파트의 선행 작업을 기다리는 의존성

완료됐다고 명시된 일, 단순 현황 공유, 근거 없는 아이디어, 실행 주체와 다음 행동이 모두 불명확한 문장은 바로 체크박스로 만들지 않는다.

### 3. Jira 최신 맥락 대조

명시된 Jira 키를 먼저 읽고, 이어서 핵심 명사와 사용자 결과로 MAIT-PM의 관련 티켓을 검색한다. 다음 정보를 함께 판단한다.

- 요약과 설명에 적힌 목적 및 범위
- 상태와 최근 댓글
- 상위 Epic 또는 큰틀, 하위 작업, 연결 티켓
- 담당자와 컴포넌트
- 완료, 보류, 범위 변경, 중복 여부

**태그, 라벨, 담당자, 컴포넌트만으로 파트를 구분하지 않는다.** 회의에서 합의한 결과와 Jira가 요구하는 실제 다음 행동을 근거로 파트를 정한다. 검색 결과의 제목이 비슷하다는 이유만으로 관련 티켓이라고 단정하지 않는다.

### 4. 파트와 다음 행동 결정

대상 회의록에 이미 존재하는 파트 토글에만 배치한다. 파트는 다음 행동의 책임으로 판단한다.

- 정책, 범위, 노출 기준, 성공 조건을 정해야 하면 기획
- 정보 구조, 화면 상태, 인터랙션, 시각 결과물을 정해야 하면 디자인
- 클라이언트 동작을 구현해야 하면 FE
- 데이터, 권한, API, 집계, 서버 동작을 구현해야 하면 BE
- 검증 시나리오와 완료 확인이 필요하면 QA

하나의 주제가 여러 파트의 결과물을 요구하면 각 파트에 서로 다른 다음 행동을 만든다. 예를 들어 `LNB와 상세 대시보드의 유형별 노출 변경`은 노출 기준 확정이 필요하면 기획 할 일이고, 그 기준을 화면으로 구체화해야 하면 디자인 할 일이기도 하다. 단순히 원본 티켓에 `디자인` 태그가 없다는 이유로 디자인 작업을 제외하지 않는다.

### 5. 체크박스 작성

각 항목은 다음 표시 형태가 되도록 Notion의 네이티브 `to_do` 블록으로 작성한다.

```markdown
- [ ] <완료 여부를 판단할 수 있는 구체적인 다음 행동> (<Jira 키 또는 링크>)
```

다음 작성 규칙을 지킨다.

- 행동으로 시작하고 가능하면 한 줄로 쓴다.
- `검토`, `확인`, `정리`만 쓰지 말고 무엇을 어떤 기준까지 끝낼지 적는다.
- 관련 Jira가 있으면 키에 링크를 연결한다.
- Jira가 없더라도 지난 회의에서 명확히 합의된 행동이면 회의록 링크나 근거 문구로 추적 가능하게 한다.
- 티켓 제목을 그대로 복사하지 말고 이번 회의 전까지 필요한 다음 행동으로 바꾼다.
- 한 체크박스에 서로 독립적인 행동을 여러 개 묶지 않는다.

예시:

```markdown
기획
- [ ] LNB와 상세 대시보드의 유형별 노출 기준 및 예외 상태 확정 (MAIT-123)

디자인
- [ ] 확정된 노출 기준에 맞춰 LNB와 상세 대시보드 화면 상태 정리 (MAIT-123)
```

### 6. 기존 내용과 병합

- 기존 제목, 토글, 본문, 체크 상태, 순서를 보존한다.
- 같은 Jira 키이면서 행동도 같은 항목은 추가하지 않는다.
- 표현만 다른 동일 행동은 더 구체적인 하나로 합친다.
- 이미 완료된 체크박스는 Jira가 다시 열렸거나 새 후속 행동이 생긴 경우가 아니면 되살리지 않는다.
- 새 항목은 해당 파트 토글의 기존 미완료 할 일 다음에 추가한다.
- 적절한 기존 파트 토글이 없으면 새 토글을 만들지 말고 `미반영 검토 항목`으로 보고한다.

### 7. 쓰기와 검증

사용자가 `넣어줘`, `붙여줘`, `반영해줘`처럼 대상 페이지 수정을 명시하면 분석 후 바로 반영한다. 정리나 초안만 요청했으면 파트별 미리보기만 제공한다.

반영 후 대상 블록을 다시 읽어 다음을 검증한다.

- 모든 새 항목이 `2. 파트별` 아래 올바른 파트 토글에 있는지
- 체크박스가 중복되지 않았는지
- 기존 블록과 체크 상태가 보존됐는지
- Jira 링크와 키가 맞는지

## 결과 보고

짧게 다음 내용을 보고한다.

- 파트별 추가 또는 구체화한 할 일
- 완료 또는 중복이라 제외한 항목
- 맥락이나 파트가 불명확해 쓰지 않은 `미반영 검토 항목`
- 확인하지 못한 Notion 또는 Jira 입력

## 안전 규칙

- 이 스킬로 Jira 티켓을 생성, 수정, 이동하지 않는다.
- 관련 티켓을 모두 회의 할 일로 옮기지 않는다. 지난 회의와 연결되고 다음 행동이 필요한 것만 사용한다.
- Jira 상태만 믿고 회의의 최신 합의를 덮어쓰지 않는다. 두 소스가 충돌하면 충돌 사실을 보고하고 자동 반영하지 않는다.
- 애매한 일을 확정 업무처럼 쓰지 않는다.
- Notion 페이지 전체를 재작성하지 않는다. 대상 토글의 필요한 하위 블록만 최소 수정한다.
